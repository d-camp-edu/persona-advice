# Persona Rx — 당뇨 처방 시뮬레이터 구현 계획

## Context

`기획.md` 한 파일만 있는 그린필드 디렉토리에서 모바일 우선 웹앱(`max-width: 640px`)을 처음부터 구현한다. 제약사 담당자가 의사에게 1:1 시연하기 위한 시뮬레이션 도구로, 병원명·의사명 입력 → 환자 선택 → 5슬롯 약제 처방 → 결과 리포트 흐름이 핵심이다. 보험 삭감 로직과 비당뇨 환자 SGLT-2i 특례 같은 도메인 규칙이 깊이 들어가 있어 순수 함수로 분리해 단위 테스트로 검증한다.

### 사용자 결정 사항
- **범위**: 기획안 전체 (Admin 콘솔 7개 탭 포함)
- **데이터 저장**: 처음부터 Firebase 연동 (사용자가 firebaseConfig 제공 예정)
- **eGFR 하한선 위반**: 경고만 표시, 처방은 가능 (선택 차단 안 함)
- **기술 선택(기본 추천 적용)**: TypeScript + Zustand 상태관리, 라우팅 라이브러리 미사용(phase 기반 단일 화면 전환)

## 3. 데이터 레이어 전략

- 시드 데이터는 `src/data/seed/*.ts`에 코드로 보관. **빌드마다 자동 업로드는 하지 않는다.**
- 앱 부팅 시 `useDataStore.bootstrap()` → Firestore 7개 컬렉션을 `onSnapshot`으로 구독.
- 컬렉션이 비어 있으면 시드 데이터로 화면을 띄우지만 **자동 업로드는 안 함** (읽기 전용 fallback).
- Admin "환자 관리/약제 관리/설정" 탭의 **"기본 초기화" 버튼만** `seedRunner.uploadAll()` 호출 → Firestore에 batch write.
- 시연 화면은 항상 store만 참조하고 시드 import는 절대 직접 사용하지 않음 (seedRunner 제외).

## 4. Zustand 스토어 분할

**`useDataStore`** (마스터 데이터 + 구독)
- state: `patients[], medications[], medCategories[], drugClasses[], deductionRules[], allowedCombinations[], sideEffectExemptions[], settings`
- actions: `bootstrap()`, `subscribeAll()`, `unsubscribeAll()`, `getMedById(id)`, `getPatientById(id)`

**`useSessionStore`** (시연 진행 상태)
- state: `phase`, `rxPhase`, `hospitalName`, `doctorName`, `sessionDocId`, `currentPatientId`, `slots: (string|null)[5]`, `diagCodes`, `comorbFilter`, `lastResult`
- actions: `login()`, `selectPatient(id)`, `setSlot(idx, medId)`, `clearSlot(idx)`, `toggleDiag(code)`, `confirmPrescription()`, `resetToSelect()`, `resetToLogin()`

**`useAdminStore`** (관리자 모드)
- state: `isAuthed`, `activeTab`, 탭별 draft, `dirty`
- actions: `unlock(pw)`, `setTab(t)`, `saveDraft()`, `cancelDraft()`, `runSeedReset(scope)`

## 5. 핵심 순수 함수 시그니처

```ts
// src/lib/patientState.ts
getPatientCurrentState(
  patient: Patient,
  sessionPrescriptions: Prescription[],
  meds: Medication[]
): CurrentState

// src/lib/prescription.ts
calculatePrescription(input: {
  patient: Patient
  current: CurrentState
  slots: (Medication | null)[5]
  diagCodes: string[]
  settings: GlobalSettings
  exemptions: SideEffectExemption[]
  pastSideEffectCounts: Record<string, number>  // 위장장애 2회 이상 스킵 판정용
  rng?: () => number                              // 테스트 결정성 확보
}): PrescriptionResult

// src/lib/deductions.ts
checkDeductions(
  slots: Medication[],
  diagCodes: string[],
  currentHba1c: number,
  rules: DeductionRule[],
  allowed: AllowedCombination[],
  settings: GlobalSettings
): string[]   // 삭감 사유 메시지 배열

// src/lib/nonDmCoverage.ts
checkNonDmCoverage(
  patient: Patient,
  slot: Medication,
  settings: GlobalSettings
): { covered: boolean; reason: string }
```

`rng` 파라미터를 외부 주입 형태로 두어 부작용 발생 확률을 테스트에서 결정적으로 검증한다.

## 8. 검증 방법

### 자동 (Vitest)
- `tests/prescription.test.ts` — 시드 환자·약제 fixture 사용, `rng=() => 1.0`으로 부작용 비발생 강제
- `tests/deductions.test.ts` — E11 4 규칙 + 병용금지(DPP-4i+GLP-1) = 5개 케이스
- `tests/nonDmCoverage.test.ts` — HFrEF 통과/탈락, HFpEF 통과/탈락, CKD 통과/탈락 = 6개 케이스

### 수동 시나리오 체크리스트
1. p1 초진 + m_2 단일제 → HbA1c 7.8 → 6.6, 위장장애 부작용 가능성
2. p2 재진 + m_30 자디앙듀오 → 기획안 예시(6.8→5.6)와 일치
3. p15 LVEF=40 + m_8 포시가 → HFpEF 특례 실패 → 자가부담으로 표시
4. p18 부작용 누적 → 위장장애 2회 이상이면 부작용 스킵
5. p11 adherence='나쁨' + 단일 약제 비병포장 → HbA1c +0.4 상승

### 핵심 수정 파일 (구현 시 가장 중요)
- `src/lib/prescription.ts` — 처방 효과·부작용·병포장·순응도 계산
- `src/lib/deductions.ts` — 보험 삭감 4 규칙
- `src/lib/nonDmCoverage.ts` — 비당뇨 SGLT-2i 특례 3종
- `src/lib/firebase.ts` + `src/lib/firestoreApi.ts` — Firestore 연결·구독
- `src/store/useSessionStore.ts` — `confirmPrescription()`이 모든 로직 오케스트레이션
- `src/data/seed/medications.seed.ts` — 60종 약제 정확한 수치 입력 (오타 검토 필수)
